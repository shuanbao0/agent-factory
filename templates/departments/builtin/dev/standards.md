# 软件开发部执行标准

## 标准工作循环

```
1. 分析已有代码的调用流程和架构
2. 设计实现方案（保证代码简洁、符合逻辑、使用对应设计模式）
3. 实现方案
4. 保证测试通过
5. 更新项目文档（README / CHANGELOG / API 文档）
6. commit
```

## 开发原则

1. **先设计再编码** — 每个功能先设计方案，不直接动手写代码
2. **收入优先** — 技术选型以快速交付为核心，完美架构不如能用的产品
3. **测试驱动** — 每个功能附带单元测试，持续保持测试全部通过
4. **文档同步** — 每次 commit 前更新相关文档
5. **持续集成** — 每个功能完成即 commit，保持代码可编译可运行

## 项目知识文件

每个项目维护 `README.md` 或 `CLAUDE.md` 作为项目指南：
- 技术栈和架构概览
- 常用命令（构建、测试、部署）
- 代码组织和约定
- 当前开发进度
- 已知问题和技术债务

## 代码规范

- 所有 coding 任务必须有基本测试才能进入 review
- 新代码必须与现有架构风格一致，避免引入不兼容的模式
- API 变更必须同步更新文档
- 不在功能开发中顺手大规模重构
- 技术债务及时标记，但不在当前任务中处理

## 质量侧重

- 评审优先级：正确性 > 安全性 > 性能 > 可读性
- 必须检查可运行性和测试覆盖
- 架构变更需 PM 确认后才能开始实现

## 退出标准

每个功能模块完成时验证：
- 编译通过无 warning
- 全部测试通过（注明测试总数）
- 代码风格检查通过（lint 无错误）
- 相关文档已更新
- 功能可演示

## 协作规则

- Designer 的设计方案确认后 Backend/Frontend 才能开始编码
- Tester 在开发完成后介入，不提前编写测试用例
- 技术债务及时标记，但不在当前任务中处理

## 特殊规则

- hotfix 类任务可跳过 design 阶段，但必须补充测试
- 性能优化任务必须有 benchmark 对比数据
- 安全相关修复优先级最高，不受常规排期约束
